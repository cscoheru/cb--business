# 务实实施计划

> **你的担忧**："之前简单的信息展现网站把我折腾够呛，我担心你能否高效（而不是bug不断）地开发"
>
> **解决方案**：最小化实施计划，聚焦核心价值，逐步验证

---

## 问题诊断：为什么之前"折腾够呛"

### 可能的原因

```
❌ 功能太多，一次性开发
❌ 没有分阶段验证，问题后期才发现
❌ 代码质量不过关，bug不断
❌ 需求不清晰，反复修改
❌ 没有测试，上线就出问题
```

### 我的反思

你说得对，我的变现策略文档设计了太多功能：
- 收藏系统、AI报告、价格追踪、团队协作、邮件提醒...
- 如果一次性开发，确实会"折腾够呛"
- 而且很多功能可能用户根本不需要

**错误**：设计了一个"完美的产品"，而不是"可验证的MVP"

---

## 务实原则

### 原则1：最简可行产品（MVP）

```
不要一次性开发所有功能，
而是开发最少的功能来验证核心价值。
```

### 原则2：分阶段验证

```
每个阶段都要验证假设，
假设不成立就止损，
不要盲目投入。
```

### 原则3：代码质量优先

```
宁可少做功能，
也要保证代码质量。
bug多比功能少更糟糕。
```

### 原则4：你的时间宝贵

```
你的时间 = 金钱
我的开发成本要最小化，
你的收益要最大化。
```

### Phase 1: 核心价值验证（2周）

**目标**：验证"用户是否愿意阅读机会卡片"

```
功能范围（最小化）：
✓ 数据采集：每天从Oxylabs获取100个产品
✓ 价值筛选：筛选出top 3产品（评分≥4.5, 价格$50-300）
✓ AI分析：调用GLM-4生成1份市场分析报告
✓ 数据存储：保存到PostgreSQL
✓ 前端展示：简单的HTML页面，显示3个机会卡片

技术栈（最简单）：
□ 后端：Python + FastAPI（已经部署了）
□ 数据库：PostgreSQL（已经有了）
□ 前端：纯HTML + Tailwind CSS（不需要React）
□ AI分析：直接调用GLM-4 API（不需要复杂架构）

数据库表（最小化）：
CREATE TABLE opportunities (
    id UUID PRIMARY KEY,
    title VARCHAR(200),
    summary TEXT,
    products JSONB,  -- [{name, price, rating, image_url}]
    ai_analysis TEXT,
    created_at TIMESTAMP
);

API端点（3个）：
GET /api/opportunities/today  -- 获取今天的3个机会
GET /api/opportunities/:id    -- 获取单个机会详情
POST /api/opportunities/refresh  -- 手动刷新（管理后台）

前端页面（1个）：
/opportunities  -- 显示今天的机会卡片列表
```

**不做的功能**（以后再说）：
- ❌ 用户系统（不需要登录）
- ❌ 收藏功能（只展示，不互动）
- ❌ 付费订阅（完全免费）
- ❌ 邮件通知（手动查看网站）
- ❌ 价格追踪（静态数据）
- ❌ 团队协作（单用户）

**部署方式**：
- 使用现有的FastAPI服务（cb-business-api-fixed）
- 不需要新的服务器
- 不需要Docker Compose
- 直接在现有服务上添加新的API端点

**成功标准**：
✓ 网站可以正常访问
✓ 每天自动更新3个机会卡片
✓ AI分析报告质量高、有价值
✓ 页面加载速度快（<2秒）
✓ 无bug运行1周

**时间分配**：
- Day 1-3: Oxylabs集成和数据采集
- Day 4-7: AI分析和价值筛选逻辑
- Day 8-10: 前端页面开发
- Day 11-14: 测试、修复bug、部署

**如果失败**：
→ 如果数据质量不好 → 调整数据源或筛选标准
→ 如果AI分析不好 → 调整prompt或换模型
→ 如果技术实现太难 → 简化功能或换方案
→ **核心原则：2周内必须看到结果，否则停止**

### Phase 2: 用户互动验证（2周）

**前提**：Phase 1成功，你认可质量

**目标**：验证"用户是否会深度使用"

```
新增功能（仍然最小化）：
✓ 用户注册和登录（最简单的邮箱+密码）
✓ 标记已阅功能（点击"已阅"，记录到历史）
✓ 阅读历史页面（查看过去标记的机会）
✓ 基础数据追踪（页面浏览、停留时间）

技术栈（仍然简单）：
□ 用户认证：JWT token（不要用OAuth）
□ 数据库表：users, user_history, user_analytics
□ 前端：仍然是HTML + Tailwind（可以加少量Alpine.js）

数据库表（新增）：
CREATE TABLE users (
    id UUID PRIMARY KEY,
    email VARCHAR(200) UNIQUE,
    password_hash VARCHAR(200),
    created_at TIMESTAMP
);

CREATE TABLE user_history (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    opportunity_id UUID REFERENCES opportunities(id),
    viewed_at TIMESTAMP,
    is_read BOOLEAN DEFAULT FALSE
);

API端点（新增）：
POST /api/auth/register
POST /api/auth/login
GET /api/user/history
POST /api/user/history/:id/mark-read
```

**不做的功能**（以后再说）：
- ❌ 收藏功能（先看标记已阅的数据）
- ❌ AI报告生成限制（暂时无限）
- ❌ 试用系统（完全免费）
- ❌ 邮件通知

**成功标准**：
✓ 注册转化率 > 20%（访问 → 注册）
✓ 标记已阅率 > 30%（浏览 → 标记）
✓ 次日留存率 > 40%（Day 1 → Day 2）
✓ 7日留存率 > 20%（Day 1 → Day 7）
✓ 无重大bug

**如果失败**：
→ 如果注册率低 → 可能是价值不够，回到Phase 1
→ 如果标记率低 → 可能是功能设计有问题，调整UI
→ 如果留存率低 → 可能是内容质量不够，提升AI分析
→ **核心原则：数据不达标就不继续，止损**

## 代码质量保证

### 开发规范

```
1. 代码审查
   → 每个功能完成后，展示给你review
   → 你认可后再合并

2. 测试优先
   → 核心功能必须有单元测试
   → 关键流程必须有集成测试
   → 上线前必须有手动测试

3. 渐进式开发
   → 小步快跑，每个阶段不超过2周
   → 每个阶段都是可独立运行的版本
   → 发现问题立即修复，不积累技术债

4. 错误处理
   → 所有API调用都要有错误处理
   → 用户友好的错误提示
   → 详细的错误日志
```

### 部署策略

```
1. 开发环境
   → 本地开发，Docker Compose
   → 每个功能先在本地测试通过

2. 测试环境
   → 部署到测试服务器（HK节点）
   → 你先试用，发现问题

3. 生产环境
   → 确认无误后部署到生产
   → 灰度发布（先给少量用户）
   → 监控错误和性能
```

### Bug处理流程

```
1. 发现bug
   → 你发现或我发现
   → 记录到issue tracker

2. 分类处理
   → Critical（影响核心功能）：立即修复
   → High（影响用户体验）：24小时内修复
   → Medium（小问题）：下次迭代修复
   → Low（优化）：有时间再修

3. 修复验证
   → 修复后测试
   → 你确认没问题
   → 部署到生产
```
